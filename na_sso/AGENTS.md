# na_sso

## Purpose

FastAPI application for administering local users and propagating their credentials to enabled external targets.

## Ownership

Owns the Python application package, persistence model, routes, synchronization, bundled web assets, and optional demo target service. Connector-specific contracts belong to `connectors/`; mock API contracts belong to `mock_targets/`; page markup belongs to `templates/`; the copied design kit keeps its contract in `static/design/`.

## Local Contracts

- `main.py` is the application entry point and registers every route module.
- Bootstrap settings come from `NA_SSO_*` environment values; target instances and password/SSH-key policies come from the YAML file named by `NA_SSO_CONFIG_FILE`. YAML secrets use exact `${ENV_NAME}` references; never commit live credentials.
- Managed-user support guidance and its optional safe http/https/mailto path come from `support_policy`; user-facing access views expose canonical state and credential mode but never management endpoints, credentials, or raw connector detail.
- Plaintext account passwords are never persisted. Initial, admin-reset, and restore passwords remain local-only temporary credentials: assigned targets stay uncreated or disabled in `CHPW` until the user changes the temporary password. Only that user-chosen replacement is Fernet-staged for propagation and cleared after assigned targets consume it; `awaiting_credentials` is not retried until verified authentication or a password action supplies a credential.
- Expired-password acknowledgement follows the configured disabled, full-renewal, or shorter-grace mode and per-password limit. It records the effective extension separately from the original password-change timestamp; only an actual password change resets acknowledgement history.
- Generated passwords require a server-validated handoff acknowledgement before account creation, reset, or password change can commit. Browser-generated SSH enrollment keeps private material client-side and stores multiple named public keys. Rotation propagates the new key before revoking the selected old key; individual, emergency, and expiry revocation synchronize the complete active key set. Fingerprint, algorithm, enrolment, expiry, and truthful connector-supported last-use metadata remain visible.
- Target management credentials are entered through the admin UI, encrypted with the global secret, never rendered back, and saved and probed as one action; SSH management login accepts password, private-key, or combined password-plus-key authentication. Each revision separately records credential verification, current reachability, updated/checked/last-success times, sanitized detail, and capped automatic connection retry; propagation requires that the current revision has succeeded at least once, while a later unreachable result does not erase that credential proof.
- The immutable local root identity is displayed as `SUPERADMIN`; it never has target state, every target matrix renders `N/A`, and administrative pages expose no edit/delete controls for it. Root password changes stay on the personal account route and require the current password.
- Server-side capabilities in `permissions.py` govern every administrative route: User operators manage only ordinary users, Target operators manage target credentials and health, Auditors investigate/export events, and the immutable Root security administrator assigns roles and retains every capability. Navigation is only a reflection of these checks. Legacy `admin` rows migrate to User operator; non-root operators cannot assign roles or mutate other operator accounts.
- State-changing browser requests reject hostile Origin/Referer and Fetch Metadata before routing; session, feedback, and MFA cookies are HttpOnly, Strict SameSite, and optionally Secure behind HTTPS. Every response denies framing and content sniffing and uses a same-origin referrer policy; HTTPS responses advertise HSTS.
- Operator-only MFA in `mfa.py` can require WebAuthn with user verification or replay-protected TOTP without changing ordinary-user or target login. MFA enrolment/revocation and recovery-code replacement require a fresh password confirmation; recovery codes are hashed and one-use, TOTP secrets are encrypted, passkey public material/sign counts are persisted, required final factors cannot be revoked, and the optional environment-owned root emergency code is one-use per configured value and audited.
- Outbound notifications in `notifications.py` are opt-in YAML preferences. Persistent sync failure, password expiry, terminal lifecycle, explicit broad/destructive approval, and access-review reminders enqueue deduplicated allowlisted payloads only for subscribed destinations. The durable queue signs `timestamp.body` with endpoint HMAC-SHA256, caps exponential retry, persists only safe failure classes/status, records terminal audit evidence, and obeys both config and root-controlled runtime disable state; endpoint secrets are never rendered.
- Managed usernames are 1–64 lowercase letters, digits, underscores, dots, or hyphens; separators may not be the first or last character. Connector preflight may apply stricter target-specific rules.
- `lifecycle.py` is the single source of truth for lifecycle commands, operation statuses, sync-state values, transition conflicts, terminal checks, and user-facing state presentation. Routes, workers, and templates must consume those contracts rather than inventing values or labels.
- User changes fan out only to assigned stable target IDs through `sync.py`; unassignment disables, removed/ambiguous targets retain retired history, and every lifecycle request, target attempt, sync state, and audit event carries a durable operation correlation ID.
- Read-only discovery uses `reconciliation.py` to classify each desired-versus-actual identity, profile, status, membership, and public-key field as match, drift, unknown, or unsupported. Reports contain sanitised observations only; connector reads must never mutate remote state.
- Target-local account discovery in `unmanaged.py` is bounded, exclusion-aware, and read-only. Ignore decisions persist; adoption links without remote mutation and gates sync on credential handoff. Remote deletion is configuration-disabled by default and requires Root recovery acknowledgement, saved sanitised evidence, a separate exact confirmation, and a one-use approval.
- Reconciliation previews in `reconcile.py` are durable and bounded. Manual and scheduled discovery never mutate; scheduled unknown reads use configurable capped exponential retry and never self-approve. Every repair requires a saved one-use approval token, destructive remote deletion requires a separate confirmation, and execution is idempotently correlated through a parent reconciliation operation and per-user child lifecycle operations.
- Bulk assignment/onboarding, unassignment/offboarding, disable, and retry accept at most 100 deduplicated account IDs, require an explicit preview and confirmation, persist a parent correlation result with partial validation detail, and treat its UUID replay token idempotently.
- CSV/JSON bulk onboarding and offboarding in `bulk.py` accepts at most 1,000 rows, maps saved target IDs, persists a no-change validation preview, and requires the same actor and idempotency key for execution. Rows succeed independently under parent/child operations; CSV/JSON outcome exports never contain credentials, while generated temporary passwords are encrypted until one audited POST download and then erased.
- Assignment profiles in `assignments.py` are immutable published versions of target and group/role bundles. Applying a version requires a saved preview; existing direct targets become visible include exceptions. Per-user include/exclude and membership add/remove exceptions are stored separately, survive later profile reconciliation, and resolve into the exact membership intent used by both sync and read-only drift comparison.
- Account lifecycle governance in `governance.py` records required owner/reason, optional UTC start/end, temporary status, inactivity review timing, and an explicitly approved disable/delete end action. Successful local authentication refreshes activity; the scheduler applies due transitions through normal lifecycle operations, opens deduplicated inactivity reviews, and audits reminders.
- Access reviews begin as no-change campaign previews, open through a one-use token, snapshot owner/reason, and accept one idempotent reviewer attestation per account. Retain reschedules review; disable/delete decisions use correlated lifecycle operations, and delete requires a separate confirmation.
- The versioned automation surface in `api.py` reuses central permissions, inventory, target readiness, operations, bulk, reconciliation, and audit services. `api_contract.py` owns JSON errors, request IDs, per-principal rate limits, pagination metadata, and durable actor/method/path/fingerprint-bound idempotency. API serializers are explicit allowlists and never expose password/key material, pending secrets, target addresses/credentials, or unredacted connector detail.
- Service accounts in `service_accounts.py` are Root-created API-only principals with an explicit subset of users, targets, and audit capabilities; they can never receive security-management authority or a browser session. Bearer credentials use the `nas_<prefix>_<random>` format, are stored only as domain-separated keyed hashes, expire, rotate with overlap, revoke independently or with the parent account, and render plaintext only on the non-cacheable issuance response. `cli.py` remains a thin API v1 client, supports secret-file input so tokens need not enter shell history, never prints a token, and never reimplements server validation or lifecycle rules.
- Authenticated sync-state SSE is served by `status.py`; initial HTML and streamed updates use the same presentation payload, and event payloads never include credentials or pending secrets.
- Redirecting mutations use signed, short-lived one-time feedback from `feedback.py`; template errors preserve only safe non-secret values, render a focused accessible summary, and never repopulate passwords or private keys.
- Lifecycle mutations are serialized per user. Delete supersedes unfinished non-delete work and applies from every state, including `CHPW`; duplicate delete and failed-operation retry reuse their operation; other concurrent mutations are rejected.
- Failed target operations persist attempt and retry timing metadata; the single-process recovery worker replays the user's persisted desired action under the owning operation.
- Deletion is soft locally: only terminal remote deletion sets `deleted_at`; restore and guarded purge require that terminal result. Restore requires a new local temporary password and returns assigned targets directly to `CHPW` because their remote accounts are already absent.
- Delete, restore, and purge use dedicated confirmation pages. Delete progress exposes its correlation prefix, completed/total counts, failures, and named blocking targets; restore and purge links render only after terminal remote deletion, and purge requires exact username confirmation.
- SQLite access goes through `db.py`; schema entities live in `models.py`.

## Verification

- Run `.venv/bin/pytest -q` from the repository root after application changes.
- For container-affecting changes, use `./compose-helper.sh rebuild` and bounded helper logs from the repository root.

## Feature Map

- **Application configuration and startup** — Loads strict YAML target/policy configuration with environment-backed secrets, initializes the database, bootstraps the admin account, mounts the NA-SSO connected-identity logo/favicon set and responsive icon-bearing collapsible-sidebar, page-rhythm, and compact-footer adaptations, and registers routers. Start: `config.py`. Files: `main.py`, `db.py`, `models.py`, `static/na-sso-logo.png`, `static/favicon.svg`, `static/favicon.ico`, `static/apple-touch-icon.png`, `static/icon-192.png`, `static/icon-512.png`, `static/site.webmanifest`, `static/app.css`.
- **Local authentication, administrator MFA, and account security** — Provides same-origin request enforcement, hardened role-aware versioned signed-cookie sessions and response headers; policy-driven operator-only WebAuthn/TOTP challenges; hashed one-use recovery and protected root emergency recovery; fresh-password MFA changes; password accept/change workflows with explicit expiry renewal/grace limits and audit evidence; policy/history enforcement; and browser-first public-key-only SSH enrollment. Start: `auth.py`. Files: `request_security.py`, `mfa.py`, `security.py`, `config.py`, `models.py`, `db.py`. Detail in `./templates`.
- **Managed-user lifecycle** — Creates managed-user and scoped-operator local accounts under the shared cross-target username contract; updates, resets, disables, soft-deletes, restores, purges, and retries them through typed conflict-safe transitions while enforcing immutable root invariants. Start: `users.py`. Files: `lifecycle.py`, `operations.py`, `models.py`, `security.py`, `sync.py`. Detail in `./templates` and `./connectors`.
- **Scoped delegated administration** — Centrally maps User operator, Target operator, Auditor, and protected Root security administrator roles to server-enforced capabilities, safe default landing routes, role-aware navigation, root-only audited assignment, and legacy-admin migration. Start: `permissions.py`. Files: `auth.py`, `users.py`, `status.py`, `audit.py`, `db.py`, `models.py`. Detail in `./templates`.
- **Scalable user inventory** — Provides bounded stable search, lifecycle/target/issue filtering, derived coverage and issue summaries, sorting, and pagination for administrative people views. Start: `inventory.py`. Files: `users.py`, `models.py`.
- **CSV/API bulk onboarding and offboarding** — Parses bounded UTF-8 CSV or JSON rows, validates ordinary accounts and assignment mappings without mutation, persists actor-bound idempotency, creates or updates onboarding accounts, disables offboarded assignments, correlates row operations, exports partial outcomes, and consumes generated temporary credentials through one audited download. Start: `bulk.py`. Files: `models.py`, `main.py`, `users.py`, `sync.py`, `security.py`. Detail in `./templates`.
- **Versioned assignment profiles and exceptions** — Drafts and explicitly publishes immutable target/group/role bundles, previews user application, preserves direct targets as visible include exceptions, stores per-user assignment/membership overrides separately, and resolves the selected version plus exceptions for sync and reconciliation. Start: `assignments.py`. Files: `models.py`, `users.py`, `sync.py`, `reconcile.py`, `main.py`. Detail in `./templates` and `./connectors`.
- **Lifecycle policy and access reviews** — Records account ownership/reason/activity and temporary start/end policy; schedules normal correlated enable/disable/delete transitions; opens inactivity reviews; and persists previewed campaigns, reminders, reviewer attestations, and retain/disable/delete decisions. Start: `governance.py`. Files: `models.py`, `auth.py`, `config.py`, `main.py`, `sync.py`, `notifications.py`. Detail in `./templates`.
- **Versioned automation API** — Exposes capability discovery and bounded user, target-health, operation, bulk, reconciliation, and audit workflows through shared permissions, correlated operations, redacted serializers, consistent JSON errors, rate limits, and durable request-fingerprint idempotency. Start: `api.py`. Files: `api_contract.py`, `models.py`, `config.py`, `main.py`, `inventory.py`, `bulk.py`, `reconcile.py`, `audit_query.py`.
- **Service accounts and CLI automation** — Creates least-privilege API identities; issues one-time expiring keyed-hash Bearer credentials; supports overlap rotation, last-use metadata, individual/account revocation, and audited controls; and scripts preview/apply/status/export workflows without duplicating domain logic. Start: `service_accounts.py`. Files: `cli.py`, `api_contract.py`, `api.py`, `models.py`, `config.py`, `main.py`. Detail in `./templates`.
- **Named SSH key lifecycle** — Validates and stores multiple device keys, performs add-before-revoke rotation, individual/emergency/expiry revocation, exact desired fingerprint reconciliation, and complete authorized-key propagation while reporting unsupported last-use truthfully. Start: `ssh_keys.py`. Files: `auth.py`, `models.py`, `db.py`, `sync.py`, `security.py`, `reconciliation.py`, `connectors/ssh.py`, `config.py`. Detail in `./templates`.
- **Unmanaged-account discovery and disposition** — Enumerates sanitised target-local identities without mutation, applies protected exclusions, persists evidence and ignore/adopt decisions, and gates remote removal through Root-only two-step one-use approval. Start: `unmanaged.py`. Files: `models.py`, `config.py`, `audit.py`, `main.py`. Detail in `./connectors` and `./templates`.
- **Correlated lifecycle operations** — Persists actor/subject/action, progress, target attempts, terminal outcomes, retry ownership, and audit correlation without storing secrets. Start: `operations.py`. Files: `lifecycle.py`, `models.py`, `db.py`, `audit.py`.
- **Assignment, synchronization, and recovery** — Tracks assigned, unassigned, retired, deferred-credential, expired, and deletion states by stable target ID; serializes fan-out per user; persists capped retries; and clears staged credentials after consumption. Start: `sync.py`. Files: `lifecycle.py`, `operations.py`, `models.py`, `db.py`, `users.py`, `audit.py`. Detail in `./connectors`.
- **Desired-versus-actual reconciliation** — Defines sanitised remote snapshots and per-field comparison; persists bounded manual/scheduled previews; retries unknown scheduled reads with configurable backoff; requires one-use approval and separate destructive confirmation; and executes repair through correlated parent/child lifecycle operations. Start: `reconcile.py`. Files: `reconciliation.py`, `config.py`, `models.py`, `db.py`, `main.py`, `sync.py`, `audit.py`. Detail in `./templates` and `./connectors`.
- **Target status dashboard** — Probes enabled targets and presents their configuration and health without duplicating user synchronization state. Start: `status.py`. Files: `models.py`. Detail in `./templates` and `./connectors`.
- **Target credential onboarding** — Stores encrypted API-token, API-key/secret, administrator-password, and SSH authentication, including SSH password, private-key, and combined two-factor modes; immediately probes each revision; separates credential verification from current reachability; exposes timestamps, sanitized failure guidance, manual test, and capped automatic retry; and gates propagation on credential proof. Start: `target_credentials.py`. Files: `status.py`, `models.py`, `security.py`, `sync.py`. Detail in `./templates` and `./connectors`.
- **Live synchronization events** — Streams authenticated operation progress and shared user/target state presentation for every visible target. Start: `status.py`. Files: `lifecycle.py`, `models.py`. Detail in `./templates`.
- **Operator feedback and safe form recovery** — Carries signed one-time mutation outcomes across redirects and renders focused error summaries while preserving only non-secret form values. Start: `feedback.py`. Files: `users.py`, `auth.py`. Detail in `./templates`.
- **Audit investigation and governance** — Records administrative and connector actions; provides bounded time/actor/subject/target/action/operation/outcome filtering, friendly summaries, UTC pagination, technical detail, correlated operation drill-down, redacted paginated CSV/JSON export, and configurable event retention that preserves operation integrity. Start: `audit.py`. Files: `audit_query.py`, `audit_retention.py`, `config.py`, `operations.py`, `models.py`. Detail in `./templates`.
- **Signed outbound notifications** — Queues preference-filtered persistent-failure, expiry, lifecycle-completion, approval, and access-review reminder events as redacted deduplicated deliveries; signs outbound requests; retries with capped backoff; persists terminal outcomes; and exposes root-only disable/enable and retry controls with audit evidence. Start: `notifications.py`. Files: `config.py`, `models.py`, `main.py`, `sync.py`, `users.py`, `governance.py`. Detail in `./templates`.
- **Mock target demo** — Emulates every supported API target so the real application and connectors can demonstrate workflows without external systems, alongside isolated SSH targets from the demo Compose stack. Start: `mock_targets/app.py`. Detail in `./mock_targets`.
