# na_sso/templates

## Purpose

Jinja page templates for the NA-SSO administrative interface.

## Ownership

Owns page structure and presentation. Route behavior and template context are owned by the corresponding Python modules in the parent folder; design tokens and component classes are owned by `../static/design/`.

## Local Contracts

- Every page extends `base.html` and keeps the standard favicon, `/design/styles.css`, `/design/components.css`, and the token-only `/app.css` shell adaptation links.
- Administrative destinations live only in the capability-aware sidebar. Every destination has a distinct accessible icon and label; desktop collapse keeps the icon links visible, clickable, and natively titled while hiding only their labels. The sidebar owns the site icon and title; its desktop collapse control changes from a neutral panel glyph to a left-chevron panel glyph on hover/focus, while the collapsed site icon changes to a right-chevron panel glyph before expanding. At narrow widths it becomes an off-canvas panel opened by the brand control. The top bar contains only a keyboard-native account burger for **My account** and **Sign out**; password and operator MFA controls are centralized on **My account**. People inventories switch from the table to compact cards, and the document itself must not overflow horizontally.
- The shared `site-page` shell fills the viewport and lets `main` grow so the footer stays at the bottom on short screens without overlapping long content; standard containers expand to the available viewport width, while oversized tables scroll only when their intrinsic content no longer fits.
- Follow the lazyway.io design rules from the root contract: documented classes first, no gradients, and exactly one amber accent per page.
- Keep sidebar navigation capability-aware through `_admin_nav.html` and order destinations by administrative workflow: people and policy first, then targets and reconciliation, then service, notification, and audit operations. User operators see Users, Assignment profiles, Access reviews, and Reconciliation; Target operators see Targets; Auditors see Audit; and Root sees all of those plus Notifications and Service accounts. The shared account burger links only to **My account** and **Sign out**; individual pages must not duplicate personal links into the sidebar. Operators retain only their authorised administrative landing route. Route checks remain authoritative.
- Root-only service-account pages create scoped API identities, issue expiring credentials on a non-cacheable one-time page, expose only token prefixes and lifecycle metadata afterward, support overlap rotation and explicit revocation, and never render a token hash or previously issued plaintext. The mobile and desktop registry presentations follow the existing inventory contracts.
- The shared NA-SSO connected-identity logo/wordmark links to `/users` on administrative pages and stays within `/account` throughout regular-user profile, password, and one-time-key pages; the logo and favicon variants are served from `../static/` rather than changing the copied design kit.
- Render operator-visible connector errors safely; never render credentials or decrypted pending secrets.
- Render signed one-time mutation outcomes through `_feedback.html`; validation errors use a focusable `role="alert"` summary, preserve safe profile and selection values, and never repopulate password or private-key inputs.
- Target credential forms are write-only: never render stored values. Each disclosure separates credential verification from current reachability and shows revision, updated/checked/last-success times in UTC, sanitized inline failure guidance, connection retry/backoff, and a **Test connection** action. A failed save redirects back with that target expanded; grouped disclosures keep at most one target open. Credential replacement persists and immediately probes as one operation. Token targets require only their API token (plus Jenkins administrator username); SSH authentication selection shows and requires only password, private key, or both fields for the selected mode.
- On the Targets page, render target names at the larger standard body-text size while retaining the normal data-key colour and weight.
- Keep user synchronization state on the user detail page; the people-first Users list owns lifecycle, coverage, and issue summaries, while the Targets page owns target configuration and health only.
- Render the protected root as `SUPERADMIN` with `N/A` target cells, no live-sync cell attributes, no username edit link, and no lifecycle action controls.
- The new-user form states and mirrors the shared username contract: lowercase letters, digits, underscores, dots, and hyphens, with every separator prohibited at either edge.
- Admin-managed and self-service password forms reveal and require confirmation for manually entered new passwords; passwords created through the generator remain single-entry and use their one-time copy modal.
- Admin and personal account views show the configured password expiry date; initial/reset assignments render `CHPW` and remain uncreated or disabled until the user replaces the temporary password.
- Delete, restore, and purge use `user_action.html`, never inline password inputs or native `confirm()` prompts. Recovery links appear only after terminal deletion; restore labels and confirms the temporary password without repopulating it, purge requires the exact username, and the Users page announces correlated deletion progress and named blockers.
- Bulk assignment/onboarding, unassignment/offboarding, disable, and retry require row selection plus a no-mutation preview. Execution validates every account, reports partial outcomes and a correlation ID, and uses a one-time replay token so resubmission cannot repeat changes.
- CSV/API bulk onboarding and offboarding uses a separate saved preview for up to 1,000 rows. The UI explains the required columns and assignment separator, renders every validation/execution outcome, exports credential-free CSV/JSON results, and offers generated temporary passwords only through a one-time POST download that is visibly non-reproducible.
- Generated user passwords open a one-time modal with the complete masked selectable value, reveal and clipboard actions, and explicit saved-password confirmation. Generated account creation/reset stays disabled and is rejected server-side until that handoff is confirmed; closing without confirmation discards the generated value, while every modal close clears its plaintext source.
- Password-decision and password-change pages use the shared application shell, structured policy feedback, distinct primary change and secondary acknowledgement actions, and the same full-value reveal/copy/confirm handoff gate as managed-user creation. Expiry decisions state whether acknowledgement is a renewal or grace period, the resulting date, and its per-password count; disabled or exhausted acknowledgement is not offered.
- The personal account page is the security hub for password changes and capability-aware administrator MFA, shows identity and named SSH-key history, and gives managed users a **My access** card for every assigned target with canonical plain-language state, supported sign-in method, retry guidance, and the configured support path without raw connector or management detail. It exposes fingerprint, algorithm, enrolment, expiry, status, and truthful last-use support; offers individual and password-confirmed emergency revocation; and clearly separates password management from browser-generated enrollment and its optional fallback. Browser enrollment follows Generate → inspect fingerprint and save/copy private key → confirm → Enrol; rotation never revokes the selected old key before new-key propagation succeeds.
- The server-handled SSH fallback renders its private key once in a non-cacheable shared-shell result page with copy and file-save actions; the plaintext key is never persisted.
- Administrator MFA pages keep ordinary target sign-in out of scope: password-first login offers enrolled passkeys, TOTP, or one-use recovery; account controls describe required/optional policy, demand fresh-password reauthentication for mutations, show named passkeys and recovery count, protect the final required factor, and render TOTP setup/recovery material only once. Browser WebAuthn transforms binary fields locally and renders errors/status without injecting HTML.
- The root-only Notifications page exposes endpoint IDs/URLs, subscribed event types, signing method, runtime enabled state, and the recent durable queue at desktop and mobile widths. It never renders signing secrets or response bodies; disable/enable and terminal retry are explicit POST controls.
- Reconciliation pages expose bounded read-only account/target previews, per-field match/drift/unknown/unsupported evidence, saved correlation, safe scheduled-policy state, one-use repair approval, and a separate required destructive-deletion acknowledgement. Scheduled discovery is visibly non-mutating and never auto-approves.
- Assignment-profile pages distinguish draft preview from immutable publication, preview profile-to-user changes before confirmation, show version IDs, preserve existing direct targets as explicit include exceptions, and expose each user's effective target memberships plus durable include/exclude/add/remove overrides.
- Lifecycle-policy pages require owner and reason, label all scheduling as UTC, require temporary access to end, explain future-start disablement, and separately acknowledge scheduled remote deletion. Access-review pages distinguish draft preview from opened review, show owner/reason snapshots, record reminders and attestations, and separately confirm delete decisions.
- Every target cell uses the presentation payload owned by `../lifecycle.py` for both initial HTML and authenticated `/events/sync` updates; preserve not-assigned, unassigned, retired, credential-waiting, expiry, deletion, and retry meanings, and construct event-derived content with DOM text APIs.

## Verification

- Run `.venv/bin/pytest -q` from the repository root after template changes.
- Check rendered pages for one amber accent, valid navigation, and the required design assets.

## Feature Map

- **Application shell, feedback, and live sync** — Shared metadata and NA-SSO logo/favicon links, a distinctly iconed capability-aware collapsible/off-canvas sidebar whose destinations remain usable when collapsed, account-only top burger, consistent page rhythm, compact attributed footer, static design and application shell links, focused one-time feedback/error summary, content slot, and SSE-driven target-cell updates that consume the server's canonical state presentation. Start: `base.html`. Files: `_admin_nav.html`, `_feedback.html`, `../static/app.css`, `../static/na-sso-logo.png`, `../static/favicon.svg`, `users.html`.
- **Authentication, administrator MFA, and personal account security** — Renders credential input, a My Account security hub for password and capability-aware administrator MFA management, operator-only passkey/TOTP/recovery verification, fresh-password MFA enrol/revoke controls, one-time TOTP/recovery material, authentication errors, required password acceptance/change, explicit dated expiry renewal/grace decisions, named multi-device SSH key enrollment/rotation/revocation/history, and the non-cacheable one-time fallback key result. Start: `login.html`. Files: `mfa_login.html`, `mfa_account.html`, `mfa_totp_setup.html`, `mfa_recovery_codes.html`, `password_form.html`, `account.html`, `private_key_once.html`, `base.html`.
- **User inventory, scoped roles, and lifecycle management** — Renders a searchable/filterable/paginated people-first inventory with mobile cards, root-only described role assignment, read-only operator visibility for delegated user operators, previewed idempotent selection/CSV/API bulk actions with downloadable partial outcomes and one-time credential handoff, dedicated per-user target detail, correlated deletion progress, live password compliance, one-time modal password generation/copy, and guarded lifecycle/retry actions. Start: `users.html`. Files: `bulk_preview.html`, `bulk_import.html`, `bulk_import_detail.html`, `user_detail.html`, `user_form.html`, `user_action.html`, `_admin_nav.html`, `base.html`.
- **Target dashboard** — Shows target configuration and connector health without duplicating the Users management matrix. Start: `status.html`. Files: `base.html`.
- **Target onboarding** — Collects write-only API/admin/SSH management credentials in readiness-aware disclosure forms, dynamically showing password, private-key, or both SSH fields; saves and probes them in one action; keeps failed work expanded with inline recovery; and separately displays credential proof, reachability, revision/timestamps, manual test, and retry/backoff. Start: `status.html`.
- **Audit investigation** — Shows filterable paginated friendly events with explicit UTC, mobile cards, technical detail disclosure, page-bounded CSV/JSON export controls, and correlated operation/attempt timelines. Start: `audit.html`. Files: `operation_detail.html`, `base.html`.
- **Notification operations** — Shows root-only destination preferences and safe runtime state plus responsive recent delivery status, capped-attempt detail, disable/enable, and manual terminal retry controls without secrets or response bodies. Start: `notifications.html`. Files: `_admin_nav.html`, `base.html`.
- **Reconciliation operations** — Creates scoped read-only previews, shows durable per-field drift evidence and scheduled retry policy, requires one-use repair approval plus separate destructive confirmation, and links authorised operators to correlated outcomes. Start: `reconciliation.html`. Files: `reconciliation_detail.html`, `_admin_nav.html`, `operation_detail.html`, `base.html`.
- **Assignment profiles and user exceptions** — Drafts and publishes immutable target/group/role bundles, previews versioned user application, and shows effective assignments alongside separately stored per-user include/exclude and membership overrides. Start: `assignment_profiles.html`. Files: `assignment_profile_detail.html`, `assignment_application.html`, `assignment_exceptions.html`, `user_detail.html`, `_admin_nav.html`, `base.html`.
- **Lifecycle policy and access reviews** — Edits owner/reason/activity/start/end/temporary policy with explicit destructive acknowledgement, previews and opens review campaigns, records reminders, and captures idempotent reviewer attestations and correlated retain/disable/delete outcomes. Start: `lifecycle_policy.html`. Files: `access_reviews.html`, `access_review_detail.html`, `user_detail.html`, `_admin_nav.html`, `base.html`.
- **Service-account credential operations** — Creates least-privilege API identities, displays capability/expiry/issuer/last-use metadata, issues one-time Bearer credentials, and supports overlap rotation plus individual/account revocation without rendering stored secrets. Start: `service_accounts.html`. Files: `service_account_detail.html`, `service_account_token_once.html`, `_admin_nav.html`, `base.html`.
- **Unmanaged-account review** — Runs visibly read-only bounded target discovery, shows sanitised evidence and persistent dispositions, adopts without remote mutation, and gates Root-only removal behind recovery acknowledgement plus separate one-use execution. Start: `unmanaged_accounts.html`. Files: `_admin_nav.html`, `base.html`.
